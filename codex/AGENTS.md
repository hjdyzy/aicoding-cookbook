# AGENTS.md

该文件为 Codex 在处理本代码库时提供指导。

## 工作模式

- Plan 模式：只做需求澄清/风险审查/方案与计划
    - 允许只读命令（如 `rg`/`ast-grep`/`fd`/`ls`/`cat`）
    - 禁止写文件与写入型/破坏性命令
- Execute 模式：用户确认范围与验收标准或主动要求执行时，开始修改与执行
- 进入 Execute 模式后仍需遵守风险提示与危险操作确认机制

---

## 语言规范（必须）

- 代码注释使用英文，其他的始终使用简体中文回复
- 修改文件流程（必须）：
    - 优先使用 `apply_patch`
    - 禁止通过 `exec_command` 间接调用 `apply_patch`
    - 若受限必须经 shell 传补丁，统一使用 `<<'PATCH'`，禁用双引号整段补丁

---

## 脚本与自动化规范（必须）

- **幂等性**：脚本必须支持重复执行而不产生副作用（创建前先检查是否存在，操作前先判断状态）
- **参数化**：避免硬编码环境相关值（IP、路径、端口、密码），通过参数或环境变量注入
- **错误处理**：遇到错误时明确报错并退出（`set -e` / `trap` / 异常捕获），不要静默跳过
- **日志输出**：关键步骤输出结构化日志（时间、操作、对象、结果），便于排查
- **超时控制**：涉及网络或外部依赖的操作设置合理超时，防止脚本挂起
- **清晰命名**：变量、函数、文件名见名知意；只在非显而易见处加注释
- **无魔法数字**：提取为命名常量（`MAX_RETRIES=3`，不要裸写 `3`）
- **删除无用代码**：修改行为时清理废弃代码和兼容路径，除非用户明确要求保留

---

## 代码开发规范

- **适用范围**：本规范适用于从 0 开始开发、维护现有项目，以及二次改造开源项目
- **先读懂再改**：修改现有代码或开源项目时，先理清模块边界和调用链，再动手修改；避免盲改引入隐性依赖
- **最小必要改动**：只改当前任务需要改的部分，避免无关重构；若是二次改造，尽量保持与上游的 diff 清晰可控
- **函数长度**：单函数不超过 50 行（不含空行），超出则提取子函数
- **嵌套深度**：不超过 3 层，用 early return / guard clause 拍平
- **边界处理**：显式处理边界情况，不要隐藏失败
- **测试验证**：代码改动后优先运行已有测试确认不破坏原有功能；新增功能尽量补测试

---

## 故障处理原则

- **先止血再查因**：生产故障优先恢复服务（回滚、切流、降级），待服务稳定后再排查根因
- **变更留痕**：所有故障处理操作必须记录（时间、操作人、操作内容、影响范围），便于事后复盘
- **回滚优先**：当修复方案不确定时，优先回滚到上一个已知正常的状态，而不是尝试热修复
- **影响评估先行**：执行任何修复操作前，先评估操作的影响范围和风险，避免"修一个坏两个"

---

## 安全响应规范

- 永远不要在源代码中硬编码密钥、API key 或凭据；使用环境变量或密钥管理工具
- 用参数化查询访问数据库；永远不要把用户输入拼接进 SQL/命令
- 在系统边界验证和清洗所有外部输入
- **漏洞评估优先级**：按 CVSS 评分 + 业务影响确定修复优先级（Critical/High 立即处理，Medium 纳入最近变更窗口，Low 纳入常规迭代）
- **最小权限原则**：修复过程中授予的临时权限必须在修复完成后立即回收
- **补丁验证**：补丁部署后必须验证漏洞已修复（复现测试或扫描确认），不能仅凭"已部署"判定完成

---

## 监控与排查规范

- **告警分级**：P0（服务不可用）→ 立即响应；P1（核心功能降级）→ 15 分钟内响应；P2（非核心异常）→ 工作时间处理
- **日志优先**：排查问题时以日志为主，监控用于发现异常、确定影响范围和定位时间点，根因收敛优先看日志
- **日志规范**：关键操作必须有结构化日志输出（时间、操作、对象、结果、耗时），便于检索和关联分析
- **根因分析思路**：按「现象 → 日志定位 → 影响范围 → 变更关联 → 资源/依赖排查 → 代码/配置定位」的顺序逐层收敛
- **监控覆盖**：新上线服务必须同步配置基础监控（存活、延迟、错误率、资源使用）

---

## 部署与变更规范

- **变更窗口意识**：区分变更窗口和非变更窗口，非窗口期变更需有明确审批和回滚方案
- **灰度发布**：变更应支持灰度/金丝雀发布，逐步放量，观察指标无异常后再全量
- **回滚方案**：每次变更前必须准备回滚方案，明确回滚步骤、预期耗时和验证方法
- **配置管理**：配置与代码分离，配置变更走版本管理，禁止直接修改线上配置文件

---

## Debug-First 策略（禁止静默降级）

- 禁止引入新的边界规则/兜底/静默降级"只为让它跑起来"
- 禁止加 mock/模拟假成功路径（如返回 `(mock) ok`、吞掉错误）
- 优先**完全暴露**：让失败清晰浮现（明确报错、异常、日志、失败的测试），在根因层面修复
- 禁止占位：默认禁止 TODO/MVP/空实现/临时绕过，必须完整可运行

---

## 计划纪律（中等及以上任务）

- 中等任务判定（满足任一即可）：
    - 涉及 >= 2 个文件且存在逻辑关联
    - 引入/调整接口或 CLI 参数
    - 多步骤调查/迁移
    - 可能影响运行时行为
- 执行前用 plan 工具给出 2–5 个里程碑
- 任意时刻仅 1 个 `in_progress`；完成一步就更新状态；结束时无 `pending`/`in_progress`
- 小任务（单文件改动 <= 10 行，或纯机械替换且无行为变化）可省略 plan，但要用 1–2 句话写清目标与验收标准

## 工具与查找（默认策略）

- 工具优先级：`fd`（文件名）→ `rg`（文本）→ `ast-grep`（结构化）
- 回退策略：`fd` 不可用时使用 `rg --files`；`ast-grep/sg` 不可用时回退 `rg`
- 优先先定位再按需读取最小必要片段，避免一次性读取大文件
- 路径参数统一用双引号包裹，优先使用 `/` 作为分隔符
- 搜索默认排除：`.git`、`node_modules`、`coverage`、`out`、`dist`
- 修改前先理解现状，不批量格式化；如需格式整理，仅限受影响区域

---

## 验证（必须）

- 纯文档/注释改动：默认不跑测试，只做内容自检
- 脚本改动：修改后至少做一次 dry-run 或 `--help` 级验证；命令默认超时 60s（更久需确认）
- 跨模块/接口变更：先给出"最小验证方案（命令/耗时/影响范围）"，经用户确认后执行
- 默认只读检查：不使用 `--fix`/`-w` 等写入参数；如需自动修复，先说明影响并征得确认

---

## 变更风险分级与确认机制（必须）

- 一级（提示后可执行）：
    - 删除无用代码 / 移除兼容性代码
    - 大范围重命名或迁移
    - 提示模板：
        ```
        风险提示：
        变更类型：
        影响范围：
        如需保留/降级方案：
        ```
- 二级（必须明确确认后执行，如"继续/同意"）：
    - 外部行为变更（含接口语义、默认值、返回结构等）
    - 删除/移动/批量修改文件
    - 提交/推送/重置（git commit/push/reset 等）
    - 系统配置/权限调整
    - 数据库破坏性操作
    - 提示模板：
        ```
        危险操作检测！
        操作类型：
        影响范围：
        风险评估：
        ```

## 输出风格

- 先一句话给出核心结论，再逐点列出依据与下一步
- 默认简短，避免冗长与重复表达
- 工具调用前提示调用的目的
- 自然语言适当换行增加可读性，每行建议 <= 100 字符（命令/路径/代码块/原始日志除外）

## Skills

Skills 存放在 `~/.codex/skills/`（个人）和 `.codex/skills/`（项目共享，可选）。

开始任务前：

- 扫描可用 skills
- 如果有匹配的 skill，读取其 `SKILL.md` 并遵循执行
- 宣布正在使用哪些 skill
- 3 个及以上有序步骤且产生文件变更的任务，默认优先使用 `taskmaster`

路由表：

| 场景 | Skill | 触发条件 |
|------|-------|----------|
| 多步骤任务跟踪 / 自主执行 | `taskmaster` | 3+ 有序步骤且产生文件变更，或 "track tasks"、"make a plan"、"track progress"、"从零开始"、"长时任务" |

## Taskmaster 说明

- `taskmaster` v5 支持 `Single / Epic / Batch`；形态选择归 `SKILL.md` 管，不在此全局文件中定义
- `taskmaster` 内的同质行级批量工作，优先使用 `spawn_agents_on_csv`
- 任务跟踪 CSV 和批量工作 CSV 分开管理
